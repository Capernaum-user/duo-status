작업 결과

`remote\command-daemon.ps1`(R01 산출물)에서 검토자가 실증한 4개 결함을 수정했다. **결함 1(필수)**:
동시 실행 상한 초과 시 매 사이클 댓글을 반복 전송하던 것을 `status='queued'` 상태파일로 "안내 완료" 여부를
기억해 **이슈당 최초 1회**만 보내도록 고쳤다. 동시에 신규 명령 dedup 로직을 `status != 'queued'`일 때만
skip하도록 바꿔, 대기열 항목은 매 사이클 재평가되다가 슬롯이 비면 **자동으로 디스패치**된다(R01의 동작
보존). **결함 2(필수)**: 후속 감시 루프에서 `cmd-*.json` 파싱을 항목 단위 try/catch로 감싸 손상 파일 1개를
`remote\state\corrupt\`로 격리 후 나머지 커맨드·신규 명령 처리를 같은 사이클에서 계속하도록 고쳤다(R01은
이 파싱에 의도적으로 try/catch를 빼둔 채 AC5(사이클 생존)를 통과시킨 취약점이었다). **결함 4(필수)**:
모드 파싱 실패 시 `auto`로 fail-open하던 기본값을 `manual`로 fail-closed하고, 인식 실패를 알리는 전용
안내 댓글을 보내도록 고쳤다(`modeRecognized` 플래그로 호출부에 전달). **결함 3(선택)**: 최소 요구대로
**주석 + README 문서화만** 했다 — `Invoke-PlannerSpawn`의 실모드가 `Start-Process -Wait`로 동기 실행되어
기획 세션이 도는 수 분간 폴링 루프 전체(STOP-REMOTE 감지·다른 watching 커맨드 완료 감시 포함)가 멈추는
트레이드오프를 코드 주석과 README "알려진 한계" 항목에 명시했다. **비동기 전환은 이번 라운드에서 하지
않았다** — 미해결로 남겨둔다(아래 미해결 질문 참조).

AC5(데몬 생존) 재증명은 R01처럼 프로덕션 코드를 취약하게 두는 방식이 아니라, **저널 파일을 별도 프로세스로
독점 잠금(`FileShare::None`)해 실제 IOException을 유발**하는 방식으로 검증했다(아래 기준5 증거) — 데몬
코드는 이 지점에 원래도 try/catch가 없었고(스크립트 최상단 `$ErrorActionPreference='Stop'`이라 이 예외도
사이클 최외곽 try/catch까지 전파됨), 이번 라운드에도 그 부분은 건드리지 않았다(범위 밖).
